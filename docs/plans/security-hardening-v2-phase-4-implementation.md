# Phase 4: In-cluster Transport Encryption - Implementation Plan

## Context

Infrastructure services (Redis, PostgreSQL, RabbitMQ) run in the `infrastructure` namespace with Istio sidecar injection disabled. Istio mTLS does not apply to them. All in-cluster traffic to these services is currently plaintext, meaning a compromised pod with network access can sniff credentials and session data. Redis is the highest-value target because it carries session tokens and identity data used by ext-authz.

Phase 4 enables native TLS on each infrastructure service and configures all clients to verify the server certificate with hostname validation.

The security-hardening-v2 plan header states: **"No backward compatibility required. Rip and replace."**

---

## Architecture Decisions

### Internal CA: openssl (not mkcert)
The plan requires keeping internal service trust material separate from the browser-facing wildcard cert. A self-signed CA generated with `openssl` serves this purpose. mkcert's CA is designed for browser trust and should not be conflated with infrastructure TLS.

### Port Strategy
- **Redis**: TLS-only on port 6379 (`--tls-port 6379 --port 0`). No plaintext.
- **PostgreSQL**: SSL on port 5432. PostgreSQL negotiates SSL/non-SSL on the same port. Clients enforce via `sslmode=verify-full`.
- **RabbitMQ**: TLS-only on port 5671 (AMQPS). Plaintext 5672 disabled. Management UI stays HTTP on 15672 (internal-only).

### Local Developer Access Strategy
- Preserve the existing host-side port-forward and direct `localhost` workflows for Redis, PostgreSQL, and RabbitMQ.
- Service cert SANs must therefore cover both in-cluster DNS names and `localhost` / `127.0.0.1` so hostname-validated TLS still works for local debugging.
- Phase 4 documentation updates must replace plaintext localhost examples with TLS-aware examples instead of silently dropping the workflow.

### Client TLS Configuration
- **ext-authz (Go)**: Already has `REDIS_TLS` flag and `tls.Config{}`. Add explicit CA cert loading and set `ServerName` from the Redis host since the distroless runtime image has no system CA bundle. Code is in the orchestration repo, so changes are allowed.
- **Spring Boot services**: Use Spring SSL bundles (`spring.ssl.bundle.pem`) introduced in Spring Boot 3.2. Both services are on 3.5.7. Single `infra-ca` bundle definition, referenced by Redis and RabbitMQ SSL config. PostgreSQL uses native JDBC `sslrootcert` parameter.
- **All clients**: Mount the `infra-ca` secret as a volume at `/etc/ssl/infra/ca.crt`.

### Certificate Distribution
- Per-service TLS secrets (`kubernetes.io/tls` type) in `infrastructure` namespace for server cert+key
- `infra-ca` Opaque secret in both `infrastructure` and `default` namespaces for the CA cert
- Generated by a host-side script, following the existing `setup-k8s-tls.sh` pattern. Do not auto-run certificate generation from Tilt or containerized automation.

## Session 1: Certificate Infrastructure

**Goal**: Host-side script to generate internal CA + per-service server certs, create K8s secrets, and add a clear prerequisite gate before `tilt up`.

### New file: `scripts/dev/setup-infra-tls.sh`

Behavior:
1. Check prerequisites: `openssl`, `kubectl`, cluster reachable
2. Must be run from the host, not from Tilt automation or container startup hooks
3. Output directory: `nginx/certs/infra/` (already gitignored by `*.pem`/`*.key` rules)
4. Generate CA if not exists: `openssl genrsa` + `openssl req -x509` (CN=Budget Analyzer Infrastructure CA, 3650 days)
5. For each service (`redis`, `postgresql`, `rabbitmq`), generate server key + CSR + sign with CA. SANs for each:
   - `DNS:{svc}.infrastructure.svc.cluster.local,DNS:{svc}.infrastructure.svc,DNS:{svc}.infrastructure,DNS:{svc},DNS:localhost,IP:127.0.0.1`
6. Create K8s secrets:
   - `infra-tls-redis` (type `kubernetes.io/tls`, ns `infrastructure`)
   - `infra-tls-postgresql` (type `kubernetes.io/tls`, ns `infrastructure`)
   - `infra-tls-rabbitmq` (type `kubernetes.io/tls`, ns `infrastructure`)
   - `infra-ca` (type `Opaque`, key `ca.crt`, in BOTH `infrastructure` and `default` namespaces)
7. Idempotent: skip if cert files already exist, delete+recreate secrets

### Prerequisite gating changes
- Update `scripts/dev/check-tilt-prerequisites.sh` to validate `infra-ca` plus all three `infra-tls-*` secrets exist before Phase 4 resources are considered ready
- When the secrets are missing, fail with a clear host-side instruction: `Run ./scripts/dev/setup-infra-tls.sh on your host, then rerun the prerequisite check.`
- Do **not** add a generating `local_resource` to `Tiltfile`
- Do **not** auto-call `setup-infra-tls.sh` from `setup.sh`

### Files
- **Create**: `scripts/dev/setup-infra-tls.sh`
- **Modify**: `scripts/dev/check-tilt-prerequisites.sh`

## Session 2: Redis TLS Server

**Goal**: Enable TLS-only on Redis. Clients will break (expected; Session 3 fixes them).

Current implementation status for March 23, 2026:

- Redis is configured in-repo for TLS-only service on port `6379`; plaintext `6379` is disabled.
- The Redis Deployment mounts `infra-tls-redis` and `infra-ca`, and both probes now use `redis-cli --tls`.
- Client cutover remains deferred to Session 3, so workloads and maintenance scripts that still assume plaintext Redis are expected to fail until that session lands.

### `kubernetes/infrastructure/redis/start-redis.sh`
Change the `exec` line to:
```sh
exec redis-server \
  --appendonly yes \
  --aclfile "$ACL_FILE" \
  --tls-port 6379 \
  --port 0 \
  --tls-cert-file /tls/tls.crt \
  --tls-key-file /tls/tls.key \
  --tls-ca-cert-file /tls-ca/ca.crt \
  --tls-auth-clients no
```
- `--port 0` disables plaintext
- `--tls-auth-clients no` = server-only TLS, ACL handles auth

### `kubernetes/infrastructure/redis/deployment.yaml`
- Add volume mounts: `redis-tls` at `/tls`, `redis-ca` at `/tls-ca` (both read-only)
- Add volumes: `redis-tls` from secret `infra-tls-redis`, `redis-ca` from secret `infra-ca`
- Update probes to use TLS:
  ```yaml
  command: ["redis-cli", "--tls", "--cacert", "/tls-ca/ca.crt", "ping"]
  ```
  `REDISCLI_AUTH` env var still provides default user password for probe auth.

## Session 4: PostgreSQL TLS Server

**Goal**: Enable SSL on PostgreSQL.

### `kubernetes/infrastructure/postgresql/statefulset.yaml`

**Init container** to fix cert permissions (PostgreSQL requires key file mode 0600, but K8s secret volumes mount as root-owned):
```yaml
initContainers:
  - name: fix-tls-perms
    image: postgres:16-alpine
    command: ['sh', '-c', 'cp /tls-source/tls.crt /tls/server.crt && cp /tls-source/tls.key /tls/server.key && chmod 600 /tls/server.key && chown 70:70 /tls/*']
    volumeMounts:
      - name: postgresql-tls-source
        mountPath: /tls-source
        readOnly: true
      - name: postgresql-tls
        mountPath: /tls
```

**Main container args** (extend existing):
```yaml
args:
  - -c
  - password_encryption=scram-sha-256
  - -c
  - ssl=on
  - -c
  - ssl_cert_file=/tls/server.crt
  - -c
  - ssl_key_file=/tls/server.key
```

**Volumes**:
- `postgresql-tls-source`: secret `infra-tls-postgresql`
- `postgresql-tls`: emptyDir (writable, for permission-fixed copies)
- `postgresql-ca`: secret `infra-ca` mounted read-only at `/tls-ca`

**Probes**: No change. Localhost probes (`pg_isready -h 127.0.0.1`) don't negotiate SSL.
